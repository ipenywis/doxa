The template is a TanStack Start app on Vite, with MDX content compiled at build time and a pluggable content adapter for runtime reads.

## High-level shape

```text
Browser ──► TanStack Router ──► /docs/$.tsx (catch-all)
                                  │
                                  ├─ MDX module from import.meta.glob
                                  ├─ Sidebar    (per-section routes)
                                  ├─ Breadcrumb (per-section trail)
                                  └─ Pagination (prev/next within section)
```

Server-only routes (`/llms.txt`, `/llms-full.txt`, the AI chat agent) read content through the same `contentStore` that the build-time search indexer uses. The renderer never touches `fs` directly.

## Subsystems

### Sections

Configured in `src/settings/sections.ts`. Exactly one section is `default: true` and lives at `/<page>`; non-default sections own a folder named after their slug and live at `/<slug>/<page>`. The `<SectionTabs />` component reads this list and hides itself entirely when only the default section exists.

### Content store

`src/lib/content/store.ts` exposes an async, edge-safe API (`getAllPaths`, `getEntry`, `readRaw`, `searchContent`). The adapter is selected from `src/settings/content.ts` via `CONTENT_SOURCE` (`vite` or `github`). The agent tools (`grep`, `cat`), `/llms.txt`, `/llms-full.txt`, and the search indexer all consume `contentStore` — they never call `fs` directly. This is what makes the template runnable on Cloudflare Workers, Vercel Edge, and Docker's Node server target.

### Page routes

`src/lib/pageroutes.ts` flattens each section's nav file (`documents.json`, `documents.<slug>.json`) into a per-section route map. Helpers: `getRoutesForSection(slug)` returns the full nav (headings, spacers, items); `getPageRoutesForSection(slug)` returns just the linkable pages.

### MDX rendering

`src/routes/$.tsx` resolves the splat to an MDX module via `import.meta.glob('/src/contents/docs/**/index.mdx', { eager: true, import: 'default' })`. The available components are registered in `src/lib/components.ts`.

### Sidebar

`src/components/sidebar/index.tsx` derives the current section from the pathname (`getSectionFromPath`) and feeds the section's routes into `<PageMenu />`. `<SubLink />` renders the optional `icon:` from frontmatter via `getIcon()` from `src/settings/icons.ts`.

### Search

Build-time indexer at `src/scripts/content.ts` walks every MDX file, strips custom JSX components, builds a MiniSearch index with per-entry `section` metadata, and writes `public/search-data/index.json`. The client (`src/components/navigation/search.tsx`) hydrates the index on idle and renders section filter chips when more than one section exists.

### AI surfaces

`/llms.txt` and `/llms-full.txt` group entries by section using the same `contentStore`. The chat agent's system prompt (`src/lib/agent/system-prompt.ts`) enumerates configured sections so the agent can scope its answers.

## Adding new MDX components

<Step>
  <StepItem title="Create the component">
    Add `src/components/markdown/<your-component>.tsx`. Keep props plain — MDX users will pass JSX-like attributes.
  </StepItem>
  <StepItem title="Register it">
    Export from `src/lib/components.ts`. This is the single source of truth for what's available inside MDX files.
  </StepItem>
  <StepItem title="Document it in the playbook">
    Update `agent/component-playbook.md` (the CLI agent reads this at runtime). Cover props, variants, and an example.
  </StepItem>
  <StepItem title="Strip it from search">
    If the component wraps text content, add its name to `customComponentNames` in `src/scripts/content.ts` so the indexer focuses on the prose.
  </StepItem>
</Step>

<Note title="Keep the playbook current" type="warning">
  The playbook is consumed by the Doxa CLI agent for documentation generation.
  Out-of-date docs there produce out-of-date generated content.
</Note>
