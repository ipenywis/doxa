The Doxa assistant is intentionally narrow. It receives a system prompt with the docs file tree, uses tools to find and read documentation content, then submits the final response to a server-side scope guard.

## Tool Surface

| Tool            | Purpose                                                                             |
| :-------------- | :---------------------------------------------------------------------------------- |
| `grep`          | Search documentation content for a pattern and return file paths with line numbers. |
| `cat`           | Read the full raw MDX content for a documentation file.                             |
| `submit_answer` | Submit the final answer for scope and citation validation.                          |

Both tools read through `contentStore`, so they work with the bundled Vite source and the runtime GitHub adapter.

## Agent Workflow

<Step>
  <StepItem title="Inspect the known file tree">
    The system prompt lists all available MDX files relative to the docs root.
  </StepItem>

  <StepItem title="Search for likely matches">
    The assistant uses `grep` when it needs to locate a topic.

    ```json
    { "pattern": "content source" }
    ```

  </StepItem>

  <StepItem title="Read the relevant source">
    After finding the page, it calls `cat`.

    ```json
    { "file": "content-sources/index.mdx" }
    ```

  </StepItem>

  <StepItem title="Submit a scoped answer">
    The assistant synthesizes the result and links to relevant sections, such as `[Content Sources](/content-sources)`.
    The server accepts the answer only when its citations map to docs pages that were found through `grep` or `cat`.
  </StepItem>
</Step>

## What The Assistant Is Told

The system prompt defines strict behavior:

- Answer using only available documentation content.
- Use `grep` first when the right file is uncertain.
- Use `cat` to read full context before answering.
- Use `submit_answer` for every final response.
- Include links in the `/path#section-anchor` format.
- Include code snippets when `aiConfig.codeSnippets` is enabled.
- Classify responses as supported, partial, or unsupported.
- Return a fixed unsupported response when a question is outside the documentation scope.

<Note title="Why raw MDX matters">
  `cat` returns raw MDX, including frontmatter and component markup. That gives
  the assistant the same source material writers review in Git.
</Note>

## Tool Result Example

```text
content-sources/index.mdx:17:Two content adapters ship with Doxa: Vite and GitHub.
content-sources/index.mdx:41:Set CONTENT_SOURCE=github to fetch docs at runtime.
```

The answer should cite the relevant page instead of inventing deployment or configuration details. If the model submits a citation that was not supported by `grep` or `cat`, the guard rejects it and asks the model to try again with documentation-supported content.

## Improving Agent Answers

- Use specific h2 headings that match reader questions.
- Put key terms in frontmatter keywords.
- Keep examples complete.
- Prefer explicit configuration tables over long prose.
- Regenerate search data after content changes.
